## Contexto

O BFF expõe conteúdo de páginas via `GET /pages/:slug`, retornando JSON hardcoded que simula um SCMS.
Atualmente existem três páginas cadastradas: `home`, `product-listing` e `checkout`.

O time de produto precisa de um novo endpoint que retorne apenas os slugs disponíveis filtrados por tipo de seção.
Por exemplo: `GET /pages?hasSection=hero` deve retornar apenas as páginas que contêm uma section do tipo `hero`.

## Escopo

- Adicionar query param `hasSection` (string, opcional) no endpoint `GET /pages` do `PagesController`
- Quando `hasSection` for informado, filtrar as páginas retornadas para incluir apenas as que possuem ao menos uma section com `type` igual ao valor do param
- Quando `hasSection` não for informado, manter o comportamento atual (retorna todas as páginas)
- A lógica de filtro deve estar no `PagesService`, não no controller

## Dicas

- `PagesService.listPages()` já retorna o mapeamento de páginas — estenda-o para suportar o filtro
- O dado de sections está disponível no objeto completo da página em `PAGES`
- `class-validator` já está disponível no projeto para validar o query param

## Fora do escopo

- Não modificar o endpoint `GET /pages/:slug`
- Não alterar a estrutura do JSON de resposta de cada página
- Não criar novas páginas no objeto `PAGES`
- Não mover o conteúdo hardcoded para banco de dados ou arquivo externo

## Definition of done

- [ ] `GET /pages?hasSection=hero` retorna apenas as páginas que contêm section do tipo `hero`
- [ ] `GET /pages` sem query param retorna todas as páginas (comportamento anterior preservado)
- [ ] `GET /pages?hasSection=inexistente` retorna array vazio (não lança erro)
- [ ] A lógica de filtro está em `PagesService`, não no controller
- [ ] Há testes cobrindo os três cenários acima
