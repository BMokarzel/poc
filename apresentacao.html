Componente 01
A Task — especificação de produto, não tutorial.
tasks/backend/T-004.md
---
id: T-004   type: FEATURE   difficulty: medium
services: [core]   tags: [redis, cache]
---
## Contexto
GET /products com alta frequência. Redis no
compose mas não utilizado.
## Escopo
- Cache TTL 60s em GET /products
- Cache TTL 120s em GET /products/:id
- Invalidar cache ao criar produto
## Fora do escopo
- Não aplicar cache em orders ou customers
- Não usar Redis como fila ou pub/sub
## Definition of done
- [ ] GET /products cacheado no 2º acesso
- [ ] POST invalida cache   [ ] Testes: hit, miss
## Domain
core/src/products/products.service.ts
🎯

O QUE, não o COMO

O escopo descreve comportamento verificável. A solução técnica é decisão do dev — como em produção.

🚧

Fora do escopo previne gold-plating

Delimitar o que não deve ser feito é tão importante quanto o que deve. Avaliadores checam isso explicitamente.

✅

DoD é binário e verificável

Cada critério é sim/não. A IA busca evidência no diff — não intenção declarada.

📍

Domain como ownership

Os caminhos em Domain são propriedade da task. Outras tasks não tocam esses arquivos sem justificativa.

Componente 02
A Rubrica — critérios explícitos e critérios ocultos.

5 níveis ordenados — o primeiro que for verdadeiro vence.

Insufficient
Um ou mais critérios do DoD não atendidos
Partial
DoD no caminho certo mas com gaps em testes ou casos limite
Sufficient
Todos os DoD atendidos, nenhum critério oculto atingido
Very Good
DoD completo + ao menos um critério oculto sufficient
Exceptional
DoD + todos os ocultos + pattern file + functional check
dod_criteria

Critérios explícitos

Mapeados ao DoD da task. Cada item tem how_to_verify — onde buscar a evidência no diff. Três níveis de resultado: insufficient, partial, sufficient.

hidden_criteria

Critérios ocultos

O dev nunca os vê. São os pontos que "um dev atento perceberia" — timeout no hot path, ordem de init do SDK, serialização Mongoose. O que separa sufficient de exceptional.

tiebreaker

Peso dos critérios

Cada hidden tem peso high | medium | low. Em empate entre níveis adjacentes, critérios de maior peso decidem — garante consistência entre avaliações.

Componente 03
O Prompt — contrato imutável entre o CLI e o Claude.
System Prompt — fixo
ETAPA 1 — Avalie dod_criteria (insuf / partial / suf)
ETAPA 2 — Avalie hidden_criteria
ETAPA 3 — Verifique pattern_file e functional_check
ETAPA 4 — Determine nível final (primeiro que for true)
ETAPA 5 — Calcule skillDelta (-2 a +2)
ETAPA 6 — Gere studyPlanUpdate se houver gap
ETAPA 7 — Componha feedback personalizado
User Prompt — montado pelo CLI com {{placeholders}}
{{PROFILE_JSON}} # person.json do dev
{{TASK_MD_CONTENT}} # conteúdo do T-NNN.md
{{EVAL_JSON_CONTENT}} # rubrica do projeto
{{GIT_DIFF}} # git diff HEAD
{{TEST_OUTPUT_JSON}} # npm test output
{{FUNCTIONAL_CHECK_OUTPUT}} # make test:contract
📤

Saída: JSON puro

Claude responde apenas JSON contra o schema definido. dod_results, hidden_results, mainFeedback, profileUpdate, studyPlanUpdate.

🔒

Critérios ocultos nunca revelados

O system prompt proíbe revelar hidden_criteria. Usa o campo rationale para compor dicas sem entregar a resposta.

🔁

Profile personaliza o feedback

Se o dev tem padrão recorrente, o prompt reconhece. Se evoluiu num gap anterior, celebra — contexto acumulativo entre avaliações.

Regra de ouro: o CLI lê o arquivo de prompt verbatim. Só os {{PLACEHOLDERS}} mudam. O contrato de avaliação é estável entre todas as execuções.
