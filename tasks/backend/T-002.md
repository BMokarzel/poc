## Contexto

O BFF possui módulos para `products`, `orders` e `pages`. O time de marketing precisa de um endpoint
dedicado para banners promocionais, que hoje estão embutidos como section `promo-banner` dentro das páginas.

Banners precisam ser consultados de forma independente para renderização em componentes que não estão
associados a nenhuma página específica (ex: sidebar, header de categoria).

## Escopo

- Criar o módulo `banners` no BFF seguindo a estrutura NestJS existente (`module`, `controller`, `service`)
- Implementar `GET /banners` retornando a lista de banners disponíveis (conteúdo hardcoded, igual ao padrão SCMS do projeto)
- Implementar `GET /banners/:id` retornando um banner específico por id
- Retornar 404 quando o banner não existir
- Registrar o `BannersModule` no `AppModule`

## Dicas

- Observe como `PagesModule` está estruturado — o padrão é idêntico
- `NotFoundException` do `@nestjs/common` é o mecanismo correto para o 404
- O conteúdo dos banners pode ser hardcoded diretamente no service, como é feito em `PagesService`

## Fora do escopo

- Não conectar banners ao Core ou a nenhum banco de dados
- Não implementar criação, edição ou remoção de banners (somente leitura)
- Não reutilizar ou modificar o módulo de pages

## Definition of done

- [ ] `GET /banners` retorna array com ao menos 2 banners com campos `id`, `title` e `type`
- [ ] `GET /banners/:id` retorna o banner correto quando o id existe
- [ ] `GET /banners/:id` com id inexistente retorna HTTP 404
- [ ] `BannersModule` está registrado no `AppModule`
- [ ] A estrutura de arquivos segue o padrão dos módulos existentes (`banners.module.ts`, `banners.controller.ts`, `banners.service.ts`)
- [ ] Há testes cobrindo os três cenários acima
